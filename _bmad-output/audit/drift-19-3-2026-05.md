## Category C — screen-section feature components: canonical mapping DEFERRED to epic-19-8 (NOT design drift)

The remaining **94** component files received `// Implements: <screen-section — pending epic-19-8 mapping>` — a **NEW 4th accepted marker form** added to Rule 21 + the ESLint rule per the Sally + Amelia + Bob Party Mode (2026-05-12) ruling (the initial dev-story pass used `<utility — no .pen counterpart>` for these; Sally flagged that as a small lie because these components *do* have a design — it lives in a screen frame). These are NOT pure utilities — most implement a *section of a designed screen frame* in `ux-design.pen` (e.g. `HeroBanner` → Screen HP-1/HP-2 hero band, `ExploreBlock` → Screen HP-5, `MediaDetailPanel` → the detail screen, `LibraryGrid`/`LibraryTable` → browse screens, the `setup/*Step` files → setup-wizard screens, the `settings/*` files → settings screens, etc.). Rule 21's strict `// Implements: Component/{Name} ({pen-node-id})` form requires a *Reusable Component* node, which screen-frame sections are not — the same gray zone `ExploreBlock`/`ExploreBlocksSettings` hit in bugfix-10-6 (which deferred its canonical Rule-21 backfill to epic-19-8).

**This is explicitly NOT a list of 94 individual design-drift findings** — no `bugfix-N-X` stories are filed here. It is a single tracked batch item for **epic-19-8 (comprehensive component sweep)**: that workflow will (a) render each component, (b) locate its `.pen` screen frame / sub-section node ID via `get_screenshot` + `get_editor_state`, (c) decide per Rule 21 whether to upgrade the `<screen-section …>` placeholder to a `// Design ref: ux-design.pen Screen {ID} ({nodeId})` soft comment or a proper `// Implements:` header (if the design extracts it as a Reusable Component), and (d) classify pixel drift per Rule 22 and file real `bugfix-N` stories only for *material* drift. The 12 Category-A files serve as the worked reference examples.

Files in Category C (epic-19-8 worklist): every `apps/web/src/components/` file carrying `// Implements: <screen-section — pending epic-19-8 mapping>`. Grep to regenerate the list:

## How the backfill was applied

`scripts/backfill-rule21-headers.mjs` — idempotent one-shot: ran `eslint apps/web/src/components --format json`, took every file flagged by `local/implements-pen-node-id`, prepended the mapped header from its internal `MAPPING` table (Category A), the `<utility — no .pen counterpart>` exemption if the path was in its `CATEGORY_B` set, or otherwise `<screen-section — pending epic-19-8 mapping>` (Category C), skipping any file already starting with `// Implements:`. (Initial run used `<utility>` for B+C alike; the B/C split was applied after the 2026-05-12 Party Mode ruling.) **The script was removed in the 19-3 code-review follow-up (commit `cf10b20`) once the headers landed — it's dead code post-backfill; the auditable record of which header went on which file is the Category A/B/C tables in *this* document.**
