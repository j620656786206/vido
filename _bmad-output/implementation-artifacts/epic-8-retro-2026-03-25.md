## What Went Well

### 1. Complete Subtitle Engine Delivered — Core v4 Differentiator
Epic 8 is the largest epic to date (10 stories) and delivered the full subtitle pipeline end-to-end. This is the #1 feature that differentiates Vido for Traditional Chinese NAS users. All 10 stories completed with 100% pass rate.

### 2. SubtitleProvider Interface — Stable Contract Across 3 Providers
Story 8-1 designed the `SubtitleProvider` interface once, and Stories 8-2 and 8-3 implemented it without any changes. The interface proved stable and well-designed — zero modifications needed across 3 different providers with very different integration patterns (API, web scraping, REST + JWT auth).

### 3. Content-Based Language Detection — Breakthrough Quality
Story 8-4 achieved ~99% accuracy at 1.3ms/100KB with zero external dependencies. This is the core technology that prevents false-positive simplified Chinese subtitles — the #1 pain point the product solves.

### 4. Pure Go OpenCC — Simplified Deployment
Story 8-5 chose the pure Go binding (`github.com/longbridgeapp/opencc`) over subprocess or CGo. This eliminates an external binary dependency, making Docker deployment simpler — critical for the Unraid deployment story.

### 5. Security Issues Caught by CR
Code review consistently found real security issues: OOM risks (unbounded io.ReadAll), SSRF in Zimuku, API key leakage in error messages. All were fixed before merge. The CR process proved its value as a security gate.

### 6. Backend-Only Stories Were Smooth
Stories 8-1 through 8-7 and 8-10 were all backend-only. They completed cleanly with single-round CR. The pattern: focused scope + single stack = predictable delivery.

## What Could Be Improved

### 1. Story 8-8 Was Too Large (XL) — 3 Rounds of CR
Story 8-8 (Manual Subtitle Search UI) was the only story spanning both backend and frontend with significant work on both sides:
- Backend: 3 new handlers + engine.go modification + CN conversion policy
- Frontend: service + hook + component + integration + 24 tests
- 12 tasks, 11 acceptance criteria
- Pass 1: 5 Critical + 8 Medium + 4 Low
- Pass 2: Integration fixes
- Pass 3: 3 High + 4 Medium + 2 Low

**Root Cause:** Cross-stack stories with heavy work on both sides create too many interaction points. The dev agent completed core tasks but left several tasks unchecked (7.5, 9.5-9.6, 10, 12), which Alexyu discovered when reviewing the story file.

**Lesson Learned:** Stories that span backend + frontend with substantial work on both sides MUST be split. The splitting rule: if both backend and frontend have >3 tasks each, it's two stories.

### 2. Dev-Story Completion Verification Failed
The dev agent marked Story 8-8 as done without verifying all task checkboxes were completed. This violated Team Agreement #1 from Epic 1: "標記完成 = 驗證完成".

**Root Cause:** Dev-story workflow lacks a mandatory final step that cross-checks every `[ ]` in the story file before declaring completion.

**Action Required:** Add explicit task-checkbox verification as the final step in dev-story workflow.

### 3. Epic 7 Retro Action Items — 1 out of 6 Completed
Only 1 of 6 action items from Epic 7 retro was completed (UX verification before CR). The remaining 5 were not addressed:
- Fix husky pre-commit hook → formatting issues recurred in Epic 8 CI
- Add EventScanComplete/Cancelled → still missing
- Extract shared scanner test fixtures → not done
- Replace auto-dismiss setInterval → not done
- Document lazy SSE pattern → not done

**Root Cause:** Retro action items lived only in the retro document. No tracking mechanism existed to surface them during next epic's sprint planning.

**Action Required:** ALL retro action items (regardless of priority) must be created as tracked entries in sprint-status.yaml. This is now a process rule going forward.

### 4. Recurring Security Patterns in CR
The same categories of security issues appeared across multiple stories:
- **Unbounded I/O (OOM risk):** 8-1, 8-2 — `io.ReadAll` without `io.LimitReader`
- **Thread safety / concurrency:** 8-2, 8-5, 8-9 — race conditions, mutex issues
- **Context lifecycle:** 8-9 — request context used for background goroutine (batch cancelled after 1 item)

**Lesson Learned:** These are systemic patterns, not one-off mistakes. Consider adding them to project-context.md as explicit anti-patterns with examples.

### 5. CI Formatting Issues Persist
Prettier format check failures appeared at the end of Epic 8, indicating husky pre-commit hooks are still not catching formatting issues locally. This was Epic 7 Action Item #1 (unfixed).

## Key Technical Decisions

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| SubtitleProvider interface in 8-1 | Stable contract for 3 providers | Zero modifications needed — excellent design |
| Content-based detection (not filename) | Filename labels are unreliable from fansub groups | ~99% accuracy, eliminates false positives |
| Pure Go OpenCC (not CGo/subprocess) | Simpler deployment, no external binary | Works perfectly, ~50ms/100KB |
| Sequential batch processing (not parallel) | Simplifies rate limiting, avoids provider throttling | 3s delay between items provides natural rate limiting |
| CN content policy (production_countries) | Mainland content should keep simplified subtitles | Implemented in 8-8 with user-override toggle |
| SSE for progress (not polling) | Real-time updates for batch/download progress | Reuses Epic 7 SSE infrastructure |
