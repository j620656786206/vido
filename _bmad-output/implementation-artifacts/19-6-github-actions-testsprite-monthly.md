## Story

As a maintainer of the Vido TestSprite integration (Free plan, 150 credits / month),
I want a GitHub Actions cron workflow that — on the 1st of every month at 03:00 UTC — reads a commit-tracked queue file, runs the next batch of TestSprite test cases up to an 80 %-quota ceiling (120 of 150 credits, reserving 30 for ad-hoc), and commits the updated queue progress back to the repo,
so that (a) TestSprite credits stop expiring unused at month-end (the 2026-03-15 deferral memo notes credits silently rolling over and getting wasted), (b) the 62-case journey suite gets continuous monthly coverage instead of ad-hoc post-deploy runs, (c) Vido's drift-prevention story (Rule 22 visual diff via 19-4/19-5 + journey diff via TestSprite) has the journey half running in CI without needing a local cron on a developer's machine, and (d) 19-7's month-end quota-warning workflow has a structured `testsprite-queue.yaml` audit trail to read instead of having to scrape TestSprite's account-info API alone.

## Acceptance Criteria

1. [@contract-v1] **A new GitHub Actions workflow file `.github/workflows/testsprite-monthly.yml` exists** with exactly three triggers: (a) `schedule: - cron: '0 3 1 * *'` (1st of month, 03:00 UTC — chosen because it's the lowest cron-traffic hour for `github.com` and falls before most engineering teams' Monday; TestSprite's quota window is calendar-month per their billing — first-of-month is when the previous month's unused credits expire and the new month's 150 are issued, so consuming early in the new window maximises lead time for follow-up reblesses); (b) `workflow_dispatch` (manual trigger for testing the workflow itself OR mid-month catch-up runs); (c) NO `push` / `pull_request` triggers — the workflow MUST NOT run on every push (would burn quota at a rate of N pushes / month × ~5 credits / case). Workflow `name:` is `TestSprite Monthly`. Single job `name: TestSprite Monthly / Consume`. `runs-on: ubuntu-24.04` (pinned per the 19-5 AC #5 rationale — deterministic runner image, do NOT use `-latest`).

2. [@contract-v1] **A new commit-tracked file `_bmad-output/audit/testsprite-queue.yaml` exists** as the workflow's single source of truth for what to run when. Schema (YAML 1.2, top-level keys ONLY in this order — workflow MUST preserve order on writes):

   ```yaml
   # _bmad-output/audit/testsprite-queue.yaml
   schema_version: 1                   # bump on any schema-shape change; downstream 19-7 reads this
   quota:
     monthly_budget: 150               # Free-plan ceiling; bump if user upgrades
     consumption_cap_pct: 80           # this workflow's self-imposed ceiling — never crosses
     reserved_credits: 30              # = monthly_budget - (monthly_budget * consumption_cap_pct / 100); kept in-sync explicitly
   last_run:
     started_at: null                  # ISO 8601 UTC; null = never run
     finished_at: null
     credits_at_start: null            # integer remaining from testsprite_check_account_info
     credits_at_end: null
     credits_consumed: null            # = credits_at_start - credits_at_end
     test_ids_run: []                  # ordered list of testsprite TC IDs (e.g. ["TC001", "TC003", ...])
     status: null                      # "success" | "budget-exhausted" | "api-failure" | "test-failures-only"
     run_url: null                     # GitHub Actions run URL — back-link for triage
   queue:
     # Ordered list — entry [0] runs next. Workflow rotates: after a successful run, the consumed
     # entries are moved to `history` (below) and the next batch picks up from the remaining queue.
     # When the queue is empty, the workflow refills from `history` (oldest-first, simple FIFO rotation —
     # this is the "continuous coverage" mode mentioned in the user story).
     - id: TC001
       file: testsprite_tests/TC001_Dashboard_loads_core_panels_and_status.py
       credits_estimated: 5            # informational; workflow uses LIVE account-info, never trusts this
       last_run_at: null
       last_status: null               # "pass" | "fail" | "error" | null (never run)
       last_run_url: null
     # ... 62 entries total seeded from `ls testsprite_tests/TC*.py` at story-create time
   history: []                         # entries pop from `queue` to here after each run; cap at 200 (prune oldest beyond)
   ```

   The file MUST be created by THIS story with all 62 test cases seeded (one entry per `testsprite_tests/TC*.py` file existing at story-implementation time), `last_run_at: null` / `last_status: null` on every entry, `last_run: { started_at: null, ... }`. Seed `credits_estimated: 5` uniformly (TestSprite's actual per-case credit cost is opaque from outside; 5 is conservative — at 5/case, 24 cases fit the 120 budget; the workflow uses the LIVE `testsprite_check_account_info` API to decide when to stop, so the estimate is informational only).

3. [@contract-v1] **Budget enforcement uses LIVE account-info, not the queue's estimate.** The workflow MUST call `testsprite_check_account_info` (via the TestSprite MCP package CLI — see Tasks) BEFORE each test case run, comparing `remaining_credits` against `reserved_credits` (30 from AC #2). When `remaining_credits ≤ reserved_credits`, the workflow stops the consumption loop, sets `last_run.status: "budget-exhausted"`, and falls through to the commit-back path (AC #4). When `remaining_credits > reserved_credits`, it runs the next queue entry. The workflow MUST NOT consume more than `consumption_cap_pct%` of `monthly_budget` in one execution (e.g. for the default 80 % / 150 = 120 credits ceiling — once `credits_at_start - remaining_credits ≥ 120`, stop). Both ceilings are honoured (whichever hits first); the dual check guards against (a) someone manually consuming credits between cron runs (the 30-reserve floor catches it) and (b) a runaway test that consumes far more than its 5-credit estimate (the 120-batch ceiling catches it). The exact account-info JSON shape is documented in TestSprite's MCP docs — the workflow MUST be defensive about field naming (`credits_remaining` vs `remaining_credits` vs `available_credits` — verify the actual key during implementation; if the API ever changes shape, the workflow fails closed by default rather than ignoring the check).

4. [@contract-v1] **Commit-back format.** After each run (success OR budget-exhaust OR partial test-failure — see AC #6 for what counts as workflow failure), the workflow MUST update `_bmad-output/audit/testsprite-queue.yaml` in place — moving consumed entries from `queue` to `history` (cap `history.length` at 200, prune oldest beyond), updating each consumed entry's `last_run_at` / `last_status` / `last_run_url`, and rewriting the `last_run:` block — then commit via the workflow's own bot identity:

   ```
   chore(testsprite): monthly run {YYYY-MM} — {N} cases consumed, {credits} credits
   
   Status: {success | budget-exhausted | test-failures-only}
   Test IDs run: TC{xxx}, TC{xxx}, ...
   Credits: {start} → {end} (Δ {consumed})
   Run URL: {github-actions-run-url}
   
   Auto-generated by .github/workflows/testsprite-monthly.yml
   
   Co-Authored-By: github-actions[bot] <41898282+github-actions[bot]@users.noreply.github.com>
   ```

   Push directly to the workflow's default branch (`main`) — this is NOT a PR (mechanical update, no review value; user can revert if catastrophic). Use the auto-issued `GITHUB_TOKEN` with job-scoped `permissions: { contents: write }` (minimum-privilege; no `pull-requests:` write needed). Git identity for the commit: `user.name "github-actions[bot]"`, `user.email "41898282+github-actions[bot]@users.noreply.github.com"`. The commit must run even if the test loop hit unhandled errors mid-run — wrap the test-loop in a try/catch (bash `set +e` around the loop), and ALWAYS reach the commit-back step via `if: always()` Actions semantics so queue mutations are never silently lost.

5. [@contract-v1] **Secrets + repo variables — exact names and minimal surface.** The workflow consumes exactly one GitHub Secret and one repo Variable; both MUST be declared in this story's Completion Notes (DEV adds them to the repo via Settings → Secrets and variables → Actions) AND in the workflow file's top-of-file comment block:
   - `secrets.TESTSPRITE_API_KEY` (Repository secret, NOT environment-scoped — TestSprite's API key is a single value per account). The workflow exports this as `TESTSPRITE_API_KEY` env var on the step that invokes the CLI; the CLI picks it up via env per TestSprite MCP convention. The plaintext key MUST NEVER appear in the workflow file, in `testsprite-queue.yaml`, in commit messages, in this story file, OR in any other repo artifact — it lives only in GitHub Secrets and in the user's local untracked `testsprite_tests/tmp/config.json` (which is double-gitignored: `.gitignore:8` `testsprite_tests/tmp/` + `:74` generic `tmp/`).
   - `vars.TESTSPRITE_TARGET_URL` (Repository variable, NOT secret — it's an HTTP URL, not sensitive on its own; making it a Variable lets the user flip target between the NAS-via-tunnel and a staging deploy without code change). Default if unset: skip the workflow with a clear error (`echo "TESTSPRITE_TARGET_URL repo variable is unset; aborting — see story 19-6 AC #5 / testsprite-queue.yaml setup"`). The CLI receives this via its `--target-url` flag (or whatever the actual flag name is — DEV verifies during implementation).

   No other secrets or variables are added. The plaintext key currently in `testsprite_tests/tmp/config.json` (gitignored, present from the 2026-03-15 local bootstrap) is the SAME key the GitHub Secret must hold — the user copies it from local file → GitHub Secret value during AC #9 wiring. (If the key has been rotated since 2026-03-15, the user re-issues from TestSprite's dashboard.)

6. **Failure semantics — distinguish workflow infra failure from test failure.** A test case returning `fail` (assertion mismatch — Vido's UI broke a journey expectation) is data the workflow must record, not an Actions-level error. Workflow exit code 0 on: (a) all queued cases passed; (b) budget-exhausted before queue empty; (c) one or more test cases returned `fail` BUT TestSprite API responded successfully (the queue + history record the per-case `last_status: fail`; 19-7 / a separate alerting workflow may open Issues from this). Workflow exit code ≠ 0 (Actions UI shows ❌) ONLY on: (i) `TESTSPRITE_API_KEY` invalid or expired (`testsprite_check_account_info` fails with auth error); (ii) `TESTSPRITE_TARGET_URL` unset or unreachable; (iii) commit-back step itself fails (push rejected — typically branch protection misconfig). This bifurcation matters because the workflow runs unattended monthly — Actions email notifications fire on ❌, so we reserve ❌ for "human must intervene" not "Vido has a regression" (regressions surface via 19-7 / a future alerting story).

7. **Concurrency control.** The workflow declares `concurrency: { group: 'testsprite-monthly', cancel-in-progress: false }` — `cancel-in-progress: false` is deliberate (opposite of 19-5's `true`): a mid-run cancellation would corrupt the queue (a partial credit consume with no commit-back rollback). If a manual `workflow_dispatch` is fired while the scheduled run is in progress, the manual run queues and runs after the scheduled run completes its commit-back, OR fails fast with "another run in progress" depending on GitHub Actions' default queueing — DEV verifies the actual behaviour and documents it in the workflow's top-comment.

## Dev Notes

### Why this story exists / where it sits in epic-19

- **Epic-19's two halves:** the visual half (19-3 / 19-4 / 19-4b / 19-5) catches pixel-level drift on PR; the journey half (this story + 19-7) catches user-flow drift on a monthly cadence. The two halves attack different failure modes: visual catches CSS/layout/component-prop drift; journey catches state-machine / cross-page-workflow / API-contract drift (e.g. a search→detail navigation breaking because a route param was renamed, which a pixel diff would not catch unless the broken state happens to differ visually).
- **Why monthly, not on every PR:** TestSprite Free plan is 150 credits/month; a single journey case costs ~5 credits; a PR-time invocation against the 62-case suite ≈ 310 credits, blowing the monthly budget in one PR. The pragmatic answer is "burn the budget where it has the most signal-per-credit": one comprehensive monthly pass + a 30-credit reserve for ad-hoc post-deploy verification. If the user upgrades to a paid plan, AC #2's `quota.monthly_budget` bumps from 150 → whatever the new ceiling is and the workflow naturally scales; the 80 % cap + 30 reserve auto-recalculate. The 30-reserve floor is the **manual** integrity (ad-hoc / dev-laptop runs); the 80 % cap is the **monthly** integrity (cron doesn't accidentally lock out manual runs).
- **Why 1st-of-month at 03:00 UTC** (vs. e.g. 28th-of-month, end of cycle): consuming early in the calendar month means failed cases have 27+ days to surface in user reports / get fixed / get re-run via `workflow_dispatch` before next month's billing window resets. Consuming late (e.g. 28th) would mean a discovered regression sits unfixed across a month boundary and the next month's run still tests the broken state.
- **Why a queue file, not just "run all 62 every month":** at 5 credits × 62 cases = 310 credits, even running every case takes >2 months of budget at the 120/month cap. The queue rotates: month 1 runs cases 1-24, month 2 runs 25-48, month 3 runs 49-62 + cycles back to 1-9 (continuous coverage). Each case gets exercised every ~2.5 months on average — enough to catch slow drift without burning the budget. Per-case `last_run_at` lets 19-7's quota-warning workflow detect "case X hasn't been tested in 90 days" if we want that signal later.
- **Where 19-7 fits:** the sibling 19-7 story (`19-7-github-actions-quota-warning`, also in backlog) reads the same `testsprite-queue.yaml` at month-end (cron `0 3 28 * *`) and opens a GitHub Issue if the month had under-consumption (e.g. cron run failed silently, or manual ad-hoc consumed all 30 reserve and the cron's run completed without surfacing). 19-6 produces the structured audit trail; 19-7 reads it and alerts. The contract between them is `schema_version: 1` (this story's AC #2) — bump it ONLY if breaking the shape that 19-7 reads from.
- **Dependency:** no version-stamped upstream contract — TestSprite's external test-case files (`testsprite_tests/TC*.py`) are docs, not contract-stamped ACs; the TestSprite MCP package version (`@testsprite/testsprite-mcp@0.0.37`) is pinned in this story's Task 2 but not contract-stamped (bumps are routine; if the MCP CLI's flag names or subcommand names change in a way that breaks the workflow, that's an operational incident, not a contract violation).
