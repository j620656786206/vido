## Story

As a frontend maintainer (and any future contributor to `apps/web/src/components/` or `apps/web/src/routes/`),
I want a GitHub Actions workflow that runs the `visual` Playwright project on every PR that touches design-rendering surfaces and on every push to `main`,
so that the design-drift class of bug that hid in plain sight for months (the `HoverPreviewCard.tsx` ↔ `Component/PosterCardHover` divergence — bugfix-10-4 root cause) is caught at PR review time on a deterministic Linux runner — not after merge, not in a manual retro, not by a user report — and so that the visual harness 19-4 + 19-4b shipped (123 fixtures / 262 baselines on `-darwin`) actually starts paying drift-prevention dividends.

## Acceptance Criteria

1. [@contract-v1] **A new GitHub Actions workflow file `.github/workflows/visual-regression.yml` exists** and is invoked under exactly three triggers: (a) `pull_request` against `main` / `develop` with a `paths:` filter that fires only when one of the following changes — `apps/web/src/components/**`, `apps/web/src/routes/**`, `apps/web/src/styles/**` (Tailwind / theme tokens — global visual surface), `apps/web/tailwind.config.js`, `apps/web/src/index.css`, `apps/web/src/main.tsx`, `tests/visual/**` (spec changes or committed baseline updates), `playwright.config.ts` (the `visual` project config), `package.json` (the `test:visual*` script family), `.github/workflows/visual-regression.yml` (the workflow itself); (b) `push` to `main` / `develop` with **no** `paths:` filter — main always runs the full suite to catch drift from non-component changes (dependency bumps, transitive Tailwind changes, the runner image rolling over); (c) `workflow_dispatch` for manual re-runs. The workflow's top-level name is `Visual Regression`. The PR job is named `Visual Regression / PR` and the push job `Visual Regression / Main`.

2. [@contract-v1] **PR job is a blocking required check** — when the visual diff fails (`pnpm run test:visual` exit code ≠ 0), the PR's "Visual Regression / PR" check shows ❌ and (combined with a branch-protection rule the user/owner enables out-of-band — see Tasks) blocks merge. The job uploads the Playwright diff artifacts (`test-results/**` — failed comparisons + traces + `actual.png` / `diff.png` for each mismatched fixture) under the name `visual-regression-diffs-pr-${{ github.run_id }}` with `retention-days: 14` (long enough for review, not so long it bloats artifacts storage). The PR job MUST NOT auto-update baselines on failure — `:update` is reserved for the deliberate rebless flow (`tests/visual/README.md` §Baseline-update discipline).

3. [@contract-v1] **Workflow uses the existing `pnpm run test:visual` npm script** (NOT `nx affected -t e2e:visual` — see Dev Notes "Nx target vs npm script decision"). The script invokes `playwright test --project=visual`, which is the `visual` Playwright project pinned by 19-4 (Chromium / 1280×800 / dark / reduced-motion / `maxDiffPixelRatio 0.001` / `testMatch ['**/*.visual.spec.ts']`). The workflow MUST NOT redefine these knobs — they are owned by `playwright.config.ts` per 19-4 [@contract-v1]. Optional helper Nx target (`apps/web/project.json` → `e2e:visual` wrapping the same command) MAY be added if the implementer judges the Nx-cache benefit worth the complexity; otherwise omit and document the rejection.

4. [@contract-v1] **Linux-baseline first-run bootstrap** implements the decision tree in `tests/visual/README.md` §Platform-suffix L82–L96 (the 19-4b Task 5 Party Mode 2026-05-14 ruling): when the workflow runs and detects that **no** `*-linux.png` files exist under `tests/visual/components.visual.spec.ts-snapshots/` (i.e. the repo is in the bootstrapped-darwin-only state), the workflow MUST:
   1. Run `pnpm run test:visual:update` to generate the full `-linux` PNG set (262 files at story-19-4b close — count is informational, not contract-stamped).
   2. Open a single, separate one-off pull request titled `chore(visual): bootstrap Linux baselines` whose body (a) explains the `-darwin`-only → `-darwin` + `-linux` transition, (b) appends the audit-trail line `Linux baselines bootstrapped {YYYY-MM-DD} via {runner-image-spec — e.g. ubuntu-24.04 image label / SHA}` to `_bmad-output/audit/visual-baseline-19-4.md`, and (c) carries the GitHub label `requires-manual-review` (a Sally / UX gate per Murat's Task-5 conditional — content drift not expected, only rendering drift expected).
   3. After that PR merges, every subsequent run takes the verify-only branch (`pnpm run test:visual`) and never re-bootstraps. The first-run logic is idempotent — once `*-linux.png` files exist, the bootstrap branch is dead code from then on. Document the bootstrap pre-condition (no `-linux.png` baselines on `main`) in a top-of-workflow comment so a future maintainer can read the file alone and understand why the conditional exists.

5. [@contract-v1] **Runner-image pinning.** The workflow MUST use `runs-on: ubuntu-24.04` (a versioned label, NOT `ubuntu-latest`), because Linux font rendering — `fontconfig` / `freetype` / the rendered DejaVu / Noto / Liberation glyphs — is the deterministic-baseline lever and the `-latest` label rolls forward without a PR (per Winston's Task-5 digest-pinning concern translated into GitHub-Hosted-Runners terms). When GitHub deprecates the chosen image label (typically 6–12 months ahead of removal), the bump to the next image is its own deliberate PR: it MUST regenerate baselines via `pnpm run test:visual:update`, MUST be tagged `requires-manual-review`, MUST be its own commit per the README §Baseline-update discipline, and MUST append a new line `Linux baselines re-blessed {YYYY-MM-DD} via {new-runner-image-spec} (previous: {old})` to `_bmad-output/audit/visual-baseline-19-4.md`. The workflow file MUST carry an inline comment on the `runs-on:` line explaining "do NOT change to ubuntu-latest — see story 19-5 AC #5".

6. **Main-push job runs the full suite without path filter** (the `push` trigger above). This is a deliberate widening: PRs filter for speed (the design-drift risk surface lives under `apps/web/src/{components,routes,styles}/**`), but post-merge `main` always runs because (i) drift can also enter via dependency bumps (a React / Tailwind / Radix transitive change re-renders pixels — none of which trip the PR `paths:` filter), and (ii) the runner image itself rolls (security patches to `fontconfig` / `freetype` between AC-#5 deliberate bumps can shift Linux glyphs). A `main`-push failure is a real signal to investigate — fall-through to AC #5 deliberate-bump flow if the root cause is the runner image.

7. **Concurrency control.** The workflow declares `concurrency: { group: 'visual-regression-${{ github.workflow }}-${{ github.ref }}', cancel-in-progress: true }` so a force-pushed PR (multi-commit churn during review) auto-cancels the prior in-flight run. The main-push job MUST use a separate concurrency group key (e.g. `visual-regression-main`) so a PR cancellation never cancels a `main`-push run.

8. **No production / source-tree edits, no runtime behaviour change.** The only files this story creates/modifies are: `.github/workflows/visual-regression.yml` (new), optionally `apps/web/project.json` (one Nx target — only if AC #3's optional path is chosen), this story file, and `sprint-status.yaml` (status bump). ### Architecture / constraints — read before implementing

- **Pure CI / DevOps story. 0 Go, 0 frontend source code, 0 tests authored, 0 migrations, 0 swagger.** Cross-stack split check: backend task count = 0, frontend-source task count = 0 → single story is trivially correct (`>3 each side` threshold not met; this isn't even a "stack" boundary — it's a CI-tooling boundary).
- **Nx target vs npm script decision (AC #3): use the npm script.** The sprint-status charter for 19-5 mentions `nx affected -t e2e:visual`, but on close inspection both halves are problematic: (i) there is no `e2e:visual` target on `apps/web` today (`apps/web/project.json` declares only `test`; everything else — `serve`, `build` — is inferred by Nx plugins, but the e2e/visual surface is invoked via npm scripts not Nx targets, mirroring how `test.yml:284` calls `npx playwright test --shard=N/4` directly with no Nx wrapper); (ii) `nx affected` adds value only when there's a graph reason to skip — but the workflow's GitHub Actions `paths:` filter (AC #1) already gates the run at the trigger level, doing the same job *before* the runner spins up at all (saving ~30 s rather than ~5 s). Adding `nx affected` would mean defining `--base` / `--head` SHAs in the workflow, which is non-trivial on the `pull_request` event (the base SHA is `github.event.pull_request.base.sha` but Nx's affected detection wants a fetched ref, which means deeper checkout). For 19-5's 0–30 s budget (per sprint-status charter) the npm-script path is correct. Document the rejection inline in the workflow's top-comment so a future maintainer understands the deliberate choice. **If 19-6 / 19-7 / future epic-19 stories find an Nx-cache lever that helps the visual project (unlikely — pixel-screenshot work is not cacheable in the Nx sense), the wrapper target can be added then.**
- **Sharding decision: not for 19-5.** Sprint-status charter mentions "Main-push runs full suite (90s, 4-shard parallel)". But: 19-4b shipped the visual spec as a single `test()` block iterating the DOM-derived worklist of 123 fixtures (see `tests/visual/components.visual.spec.ts` post-Task-4 single-fixture-per-page architecture; `?manifest=1` lists fixtures, then `?fixture=<id>` runs each — but Playwright's sharding splits at the *test-file* level, not the iteration level). Sharding 1 test across 4 shards = 1 shard does all the work, 3 shards do nothing. To shard meaningfully, 19-4b's spec would have to be restructured into N `test()` blocks (e.g. per-fixture or per-`describe`). That's a 19-4b-spec change — out of scope here. Conclusion: 19-5 runs as a single job (~90 s for the main-push full suite is within budget). If a future maintainer wants real sharding, file a follow-up story to restructure the spec; do NOT inflate 19-5's scope. Document this finding so the charter ↔ reality gap is recorded.
- **Determinism on Linux vs darwin.** The committed baselines are `-darwin` (dev machine, Apple's Core Text font rendering). On `ubuntu-24.04` GitHub Hosted Runners, Chromium uses `fontconfig` + `freetype` against Ubuntu's bundled `fonts-noto`, `fonts-liberation`, `fonts-dejavu-core` set. Font hinting, sub-pixel positioning, and emoji rendering all differ. Sally's 19-4b Task 5 conclusion: "content drift unexpected (Sally already approved darwin content); only rendering drift (font hinting, emoji glyphs, sub-pixel) anticipated". The `requires-manual-review` label on the bootstrap PR is Murat's safeguard — a human eyeballs the `-linux` set before the `-darwin` baselines lose their "primary" status. Once both sets are committed, CI verifies `-linux` (the CI runner produces `-linux`) and dev verifies `-darwin` locally (Mac dev machines produce `-darwin`); the two sets co-exist permanently per Playwright's snapshot-path convention.
- **GitHub-Hosted Runners' "digest pin" equivalent (AC #5):** GitHub Actions exposes image labels like `ubuntu-22.04`, `ubuntu-24.04`, `ubuntu-latest`. There is no SHA digest you can pin. The closest thing is the versioned label, which GitHub updates only via security patches (advertised ahead of time) and rolls forward (`-latest`) only at major version boundaries (advertised 6+ months ahead via the [actions/runner-images](https://github.com/actions/runner-images) project's deprecation announcements). `ubuntu-24.04` is the right pin today (current as of 2026-05; rolls to 26.04 sometime mid-2026 per GitHub's 2-year cadence). When that bump lands, AC #5's deliberate-rebless flow kicks in.
- **Permissions: minimum-privilege.** Default `GITHUB_TOKEN` permissions in GitHub Actions are repository-defined (the repo's default is configurable). The PR verify-only job needs only `contents: read` (checkout) — no other permissions. The main-push bootstrap branch needs `contents: write` (commit the audit-doc line + the `-linux` PNGs to a new branch) + `pull-requests: write` (open the bootstrap PR via `peter-evans/create-pull-request@v6`). Scope these at the *job* level (`jobs.<job>.permissions:`) not the workflow level — minimum-privilege per project-context Rule 19's spirit. PR-job permissions block: `{ contents: read }`. Main-job permissions block: `{ contents: write, pull-requests: write }` (only because of the bootstrap path; tighten if the bootstrap is ever removed).
- **`peter-evans/create-pull-request` action choice.** This action is the de-facto standard for "open a PR from a workflow"; widely used, well-maintained, the version `v6` is current as of 2026-05. Pin to `@v6` (major-version pin per GitHub's recommendation) — DO NOT use `@main` (mutable). The action handles the branch-creation + commit + PR-open atomically. The PR body MUST be a HEREDOC-style multi-line string with the audit-trail line + a back-link to 19-5 + a one-line "What to review" callout for Sally.
- **No secrets needed.** The visual workflow does NOT need `TMDB_API_KEY` (the `/test/gallery` route renders fixtures via mocked data per 19-4 Task 2's "no network — props directly") and does NOT need `TESTSPRITE_API_KEY` (that's 19-6/19-7). The Go backend isn't even started for this workflow — Playwright's `webServer` config in `playwright.config.ts` line 167 short-circuits to `[]` when `process.env.CI === 'true'`. The visual project doesn't need a webServer at all (it navigates to `/test/gallery` which is static-routable from the production-bundle SPA dist). Confirm during implementation: does the `visual` project need `webServer` in CI? If yes, start `npx serve dist/apps/web -l 4200 --single` (mirrors `test.yml:255`) — but only if Playwright errors without it. The single test in `components.visual.spec.ts` does `page.goto('/test/gallery'...)` so SOMETHING has to serve `4200`. **The cleanest CI implementation: build the frontend (`npx nx build web --configuration=production` — but skip the backend since visual tests don't fetch real APIs), then start `npx serve dist/apps/web -l 4200 --single` in the background, then run `pnpm run test:visual`.** Mirror `test.yml`'s build → serve → playwright pattern minus the Go backend.
- **PR-blocking is half a CI plumb + half a branch-protection rule** (AC #2). The workflow makes the check appear in PR UI under the name `Visual Regression / PR`; making it *required* (i.e. block-on-fail) is a GitHub branch-protection rule the repo owner sets via Settings → Branches → Branch protection rules → "Require status checks to pass before merging" → tick `Visual Regression / PR`. The workflow can't enable this itself — but Task 5's "first PR observation" step naturally surfaces the gap (the check fires but isn't required until configured out-of-band). Document the required-check setup in the story's Completion Notes as a follow-up the user/owner does post-merge — a single web-UI click. The story's contract is the *check existing and being available to mark required*; the policy click is the user's call.
- **First-run bootstrap edge case — what if 19-5 lands BEFORE the bootstrap PR merges?** Sequence: (1) this story's PR (call it `PR-19-5`) lands the workflow file. (2) GitHub Actions starts running the workflow on PRs / pushes. (3) Main-push from `PR-19-5`'s merge fires the workflow on `main`. (4) The main job sees no `-linux.png` baselines → bootstrap branch fires → opens `PR-bootstrap-linux`.
