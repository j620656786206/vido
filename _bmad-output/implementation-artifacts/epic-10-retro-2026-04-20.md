## Challenges (Patterns from Deep Story Analysis)

### 🔴 Pattern #1: Frontend Perf + A11y is the largest CR gap

DEV consistently ships frontend code with perf or a11y issues that CR catches:

- **10-2 H1**: `getImageUrl(..., 'original')` shipped 3–5MB backdrops to mobile (violated AC #4)
- **10-2 H2**: TrailerModal declared `aria-modal="true"` but had no focus management
- **10-4 L1**: AvailabilityBadge missing `aria-live` (async appearance silent to SR)
- **10-5 H1**: Lazy-load contract conflicted with AC #4 wording ("single POST")

These aren't skill gaps — they're checklist gaps. DEV doesn't have a Frontend Perf+A11y pre-flight checklist before marking a frontend story "review".

### 🔴 Pattern #2: AC contract drift across stories (10-4 → 10-5 conflict)

- 10-4 AC #4 stated "single POST batch for all visible TMDb IDs"
- 10-5 added Intersection-Observer lazy-load → contract effectively became "≤N POSTs (one per lazy-revealed batch)"
- Murat's CR on 10-5 caught the mislabel + added an `anyEnabledInflight` stability gate

There's no mechanism today to detect when a new story's design implicitly renegotiates a prior story's AC. The fix requires retro-active grep when a story author touches an existing contract.

### 🟠 Pattern #3: Wire format Rule 7 violation (10-1a H1)

- `INVALID_YEAR_RANGE` shipped without the mandatory `TMDB_` source prefix
- Plain text grep (`{SOURCE}_{ERROR_TYPE}`) would have caught this in CR
- It was caught — but only by adversarial human review, not automation

### 🟠 Pattern #4: Story scope drift — "client method exists" ≠ "HTTP route exists"

- 10-2 Task 3.3 said "/videos endpoint already exists in client". Go client method existed; HTTP route did not. DEV had to add backend route exposure mid-story.
- 10-4 endpoint path `/movies/check-owned` was renamed to `/media/check-owned` for cross-type semantic clarity (the endpoint queries both movies + series tables).

Rule 15 self-verification covers main.go wiring + DB column sync + Swagger, but doesn't explicitly check the gap between "client method exists" and "HTTP route registered".

### 🟢 Pattern #5: NAS deployment lag (readiness gap)

Alexyu confirmed during the retro that **Epic 10 has not been smoke-tested locally yet** (let alone deployed to NAS). All test gates are green, but real-browser verification on the homepage (LCP < 2s claim from AC #2, hero pixel heights 250/400 at 390/1440, Intersection-Observer lazy-load behavior in production build) is unverified.

This is not a code regression but a workflow gap: the FULL REGRESSION GATE proves code correctness, not feature correctness.

## Key Insights

- **CR has crossed a maturity threshold.** Producing 8±1 finding per story consistently (2H/4M/2L) means CR is now load-bearing infrastructure, not a one-off audit. Treat it like a CI step.
- **Frontend perf + a11y need a dedicated checklist.** Three of four frontend stories had a perf or a11y HIGH/MEDIUM finding. This is no longer "edge-case" — it's a recurring gap that a checklist would close before review.
- **Cross-story contract evolution is invisible.** The 10-4 → 10-5 contract drift only surfaced because Murat reviewed 10-5 with 10-4's AC text in mind. Without that human check, the mislabel would have shipped.
- **"Already exists in client" is a dangerous shortcut in story tasks.** It should explicitly call out which layer ("Go client method" vs "HTTP route" vs "service interface") to avoid scope drift.
- **Tests-passing ≠ feature-verified.** 1738/1738 PASS doesn't tell you whether the homepage looks right at 390px or whether trailer playback works. Need a separate "smoke test on local dev server" gate before declaring an epic production-ready.
