- Story 4-5: `MovieRepository.Create()` missing 4 DB columns (file_path, parse_status, metadata_source, vote_average)

### Action Item #3: Series/Season/Episode Schema

Following Jellyfin/Plex industry pattern:

```
series (table)
  ├── id, title, poster_path, overview, first_air_date,
  │   last_air_date, in_production, status, networks,
  │   number_of_seasons, number_of_episodes, tmdb_id ...

seasons (new table)
  ├── id, series_id (FK), season_number, name,
  │   poster_path, overview, episode_count, air_date, tmdb_id

episodes (table — enhance back-refs)
  ├── id, series_id (FK), season_id (FK), season_number,
  │   episode_number, title, overview, still_path, file_path ...
```

Key changes to parse_queue_service:
- Branch on `mediaType`: "movie" → `models.Movie`, "tv" → `models.Series` + `models.Episode`
- Season 0 = Specials (TMDb convention)

### Research Findings

Industry survey of Netflix, Plex, Jellyfin, Emby, and TMDb confirmed:
- **3-level hierarchy (Series → Season → Episode) is the universal standard** for content catalog systems
- Netflix's 5-level NMDB architecture is for media analysis ML pipelines, not content catalogs — irrelevant to Vido
- Season should be a first-class DB entity (not JSON blob) per all major platforms
- TMDb API uses completely separate endpoints for movie vs TV (`/movie/` vs `/tv/`)
