## What Needs Improvement

### 1. JSON Naming Convention Consistency
**Problem:** Multiple stories had snake_case vs camelCase inconsistencies in JSON structures.

**Examples:**
- Story 3.9: `FilenameMapping` struct used `created_at` → changed to `createdAt`
- Story 3.12: Health response fields needed alignment

**Root Cause:** No automated linting for JSON struct tags.

### 2. Rule 11 Violations Persist
**Problem:** Interface definitions placed in wrong packages despite Rule 11 being added in Epic 2.

**Examples:**
- Story 3.9: `LearningRepositoryInterface` initially in service package → moved to `repository/interfaces.go`
- Story 3.12: Health service interfaces needed relocation

**Impact:** Required additional code review rounds to fix.

### 3. Rule 7 Violations (Missing Error Codes)
**Problem:** Generic `InternalServerError()` used instead of specific error codes.

**Examples:**
- Story 3.9: Missing `LEARNING_` prefixed error codes
- Story 3.12: Health check errors needed specific codes

**Impact:** Harder to debug and monitor production issues.

### 4. Integration Registration Gap (Recurring)
**Problem:** Same issue from Epic 2 - endpoints/services not registered in main.go.

**Examples:**
- Story 3.10: ParseProgress handler not wired
- Story 3.12: `/api/v1/health/services` endpoint missing

**Question:** Why does this keep happening despite being a known issue?

### 5. Deferred Technical Debt
**Problem:** Several stories deferred items that may accumulate:
- Story 3.9: Pattern editing capability deferred
- Story 3.10: WebSocket alternative to SSE deferred
- Story 3.6: TMDb year filter passthrough deferred

### 6. CI Verification Before Commit (CRITICAL)
**Problem:** Code was committed even when CI pipeline was not passing.

**Observed Behavior:**
- Commits pushed to repository without confirming CI success
- Failed tests or linting errors discovered after commit
- Required additional fix commits to resolve CI failures

**Root Cause:** No enforcement mechanism to prevent commits when CI is failing.

**Impact:**
- Pollutes git history with fix commits
- Other developers may pull broken code
- Undermines confidence in main branch stability
